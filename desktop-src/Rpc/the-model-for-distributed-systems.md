# <a name="the-model-for-distributed-systems"></a><span data-ttu-id="9235f-103">Modello per sistemi distribuiti</span><span class="sxs-lookup"><span data-stu-id="9235f-103">The Model for Distributed Systems</span></span>

<span data-ttu-id="9235f-104">Tradizionalmente, l'esecuzione di un sistema monolitico su più computer implica la suddivisione del sistema in componenti client e server distinti.</span><span class="sxs-lookup"><span data-stu-id="9235f-104">Traditionally, having a monolithic system run across multiple computers meant splitting the system into separate client and server components.</span></span> <span data-ttu-id="9235f-105">In tali sistemi, il componente client gestisce l'interfaccia utente e l'elaborazione back-end fornita dal server, ad esempio l'accesso al database, la stampa e così via.</span><span class="sxs-lookup"><span data-stu-id="9235f-105">In such systems, the client component handled the user interface and the server provided back-end processing, such as database access, printing, and so on.</span></span> <span data-ttu-id="9235f-106">Con la proliferazione dei computer, la riduzione del costo e la connessione da reti con larghezza di banda sempre più elevata, la suddivisione dei sistemi software in più componenti è diventata più comoda, con ogni componente in esecuzione in un computer diverso ed eseguendo una funzione specializzata.</span><span class="sxs-lookup"><span data-stu-id="9235f-106">As computers proliferated, dropped in cost, and became connected by ever-higher bandwidth networks, splitting software systems into multiple components became more convenient, with each component running on a different computer and performing a specialized function.</span></span> <span data-ttu-id="9235f-107">Questo approccio semplifica lo sviluppo, la gestione, l'amministrazione e il miglioramento delle prestazioni e dell'affidabilità, poiché l'errore in un computer non ha necessariamente disabilitato l'intero sistema.</span><span class="sxs-lookup"><span data-stu-id="9235f-107">This approach simplified development, management, administration, and often improved performance and robustness, since failure in one computer did not necessarily disable the entire system.</span></span>

<span data-ttu-id="9235f-108">In molti casi il sistema viene visualizzato nel client come cloud opaco che esegue le operazioni necessarie, anche se il sistema distribuito è costituito da singoli nodi, come illustrato nella figura seguente.</span><span class="sxs-lookup"><span data-stu-id="9235f-108">In many cases the system appears to the client as an opaque cloud that performs the necessary operations, even though the distributed system is composed of individual nodes, as illustrated in the following figure.</span></span>

![i client accedono ai servizi in un sistema di server RPC che risultano come un cloud opaco ai client esterni](images/indy-nodes.png)

<span data-ttu-id="9235f-110">L'opacità del cloud viene mantenuta perché le operazioni di elaborazione vengono richiamate per conto del client.</span><span class="sxs-lookup"><span data-stu-id="9235f-110">The opacity of the cloud is maintained because computing operations are invoked on behalf of the client.</span></span> <span data-ttu-id="9235f-111">Di conseguenza, i client possono individuare un computer ( *nodo*) all'interno del cloud e richiedere una determinata operazione. durante l'esecuzione dell'operazione, il computer può richiamare la funzionalità su altri computer all'interno del cloud senza esporre i passaggi aggiuntivi o il computer in cui sono stati eseguiti nel client.</span><span class="sxs-lookup"><span data-stu-id="9235f-111">As such, clients can locate a computer (a *node*) within the cloud and request a given operation; in performing the operation, that computer can invoke functionality on other computers within the cloud without exposing the additional steps, or the computer on which they were carried out, to the client.</span></span>

<span data-ttu-id="9235f-112">Con questo paradigma, i meccanismi di un sistema distribuito di tipo cloud possono essere suddivisi in molti singoli scambi di pacchetti o conversazioni tra i singoli nodi.</span><span class="sxs-lookup"><span data-stu-id="9235f-112">With this paradigm, the mechanics of a distributed, cloud-like system can be broken down into many individual packet exchanges, or conversations between individual nodes.</span></span>

<span data-ttu-id="9235f-113">I sistemi client-server tradizionali hanno due nodi con responsabilità e ruoli predefiniti.</span><span class="sxs-lookup"><span data-stu-id="9235f-113">Traditional client-server systems have two nodes with fixed roles and responsibilities.</span></span> <span data-ttu-id="9235f-114">I sistemi distribuiti moderna possono avere più di due nodi e i loro ruoli sono spesso dinamici.</span><span class="sxs-lookup"><span data-stu-id="9235f-114">Modern-distributed systems can have more than two nodes, and their roles are often dynamic.</span></span> <span data-ttu-id="9235f-115">In una conversazione un nodo può essere un client, mentre in un'altra conversazione il nodo può essere il server.</span><span class="sxs-lookup"><span data-stu-id="9235f-115">In one conversation a node can be a client, while in another conversation the node can be the server.</span></span> <span data-ttu-id="9235f-116">In molti casi, il consumatore finale della funzionalità esposta è un client con un utente seduto a una tastiera, guardando l'output.</span><span class="sxs-lookup"><span data-stu-id="9235f-116">In many cases, the ultimate consumer of the exposed functionality is a client with a user sitting at a keyboard, watching the output.</span></span> <span data-ttu-id="9235f-117">In altri casi, le funzioni di sistema distribuite vengono eseguite in modo automatico, eseguendo operazioni in background.</span><span class="sxs-lookup"><span data-stu-id="9235f-117">In other cases the distributed system functions unattended, performing background operations.</span></span>

<span data-ttu-id="9235f-118">È possibile che il sistema distribuito non disponga di client e server dedicati per ogni scambio di pacchetti specifico, ma è importante ricordare che è presente un chiamante (o initiator) che viene spesso definito client.</span><span class="sxs-lookup"><span data-stu-id="9235f-118">The distributed system may not have dedicated clients and servers for each particular packet exchange, but it is important to remember there is a caller, (or initiator, either of which is often referred to as the client).</span></span> <span data-ttu-id="9235f-119">È anche presente il destinatario della chiamata (spesso definita server).</span><span class="sxs-lookup"><span data-stu-id="9235f-119">There is also the recipient of the call (often referred to as the server).</span></span> <span data-ttu-id="9235f-120">Non è necessario avere scambi di pacchetti bidirezionali nel formato Request/Reply di un sistema distribuito. spesso i messaggi vengono inviati solo unidirezionale.</span><span class="sxs-lookup"><span data-stu-id="9235f-120">It is not necessary to have two-way packet exchanges in the request-reply format of a distributed system; often messages are sent only one way.</span></span>
