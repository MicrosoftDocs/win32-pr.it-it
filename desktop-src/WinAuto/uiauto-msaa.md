# <a name="ui-automation-and-active-accessibility"></a>Automazione interfaccia utente e Active Accessibility

Microsoft Active Accessibility è l'API legacy introdotta in Windows 95 ed è stata progettata per rendere accessibili le applicazioni Windows. Automazione interfaccia utente Microsoft è il nuovo modello di accessibilità per Windows ed è progettato per soddisfare le esigenze dei prodotti Assistive Technology e degli strumenti di test automatizzati. Automazione interfaccia utente offre molti miglioramenti rispetto a Microsoft Active Accessibility. In questo argomento vengono illustrate le differenze tra le due tecnologie.

In questo argomento sono contenute le sezioni seguenti.

## <a name="programming-languages"></a>Linguaggi di programmazione

Microsoft Active Accessibility si basa sul Component Object Model (COM) con supporto per le interfacce duali e pertanto è programmabile in C/C++ e linguaggi di scripting.

Quando è stata introdotta l'automazione dell'interfaccia utente, l'API client era limitata al codice gestito, mentre l'API del provider includeva sia le implementazioni gestite che quelle non gestite. Con Windows 7 è stata introdotta una nuova API client basata su COM che semplifica la programmazione di applicazioni client di automazione interfaccia utente in C/C++.

## <a name="servers-and-clients"></a>Server e client

In Microsoft Active Accessibility, i server e i client comunicano direttamente, principalmente tramite l'implementazione del server dell'interfaccia [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible) .

In automazione interfaccia utente, un servizio di base risiede tra il server (provider) e il client. Il servizio di base effettua chiamate alle interfacce implementate dai provider e fornisce servizi aggiuntivi, ad esempio la generazione di identificatori di runtime univoci per gli elementi dell'interfaccia utente. Le applicazioni client accedono a questo servizio principale creando un oggetto [**CUIAutomation**](/previous-versions/windows/desktop/legacy/ff384838(v=vs.85)) . Questo oggetto supporta un set di interfacce client separate dalle interfacce del provider. Per ulteriori informazioni, vedere [creazione dell'oggetto CUIAutomation](uiauto-creatingcuiautomation.md).

I provider di automazione interfaccia utente possono fornire informazioni ai client Microsoft Active Accessibility e Microsoft Active Accessibility server possono fornire informazioni alle applicazioni client di automazione interfaccia utente. Tuttavia, poiché Microsoft Active Accessibility non espone quante più informazioni dell'automazione interfaccia utente, i due modelli non sono completamente compatibili.

## <a name="ui-elements"></a>Elementi dell'interfaccia utente

Microsoft Active Accessibility presenta un elemento dell'interfaccia utente come interfaccia [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible) abbinata a un identificatore figlio. È difficile confrontare due puntatori **IAccessible** per determinare se fanno riferimento allo stesso elemento.

In automazione interfaccia utente ogni elemento è rappresentato come un oggetto che espone l'interfaccia [**IUIAutomationElement**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationelement) ai client. Gli elementi possono essere confrontati in base ai relativi identificatori di runtime, recuperati usando [**IUIAutomationElement:: GetRuntimeId**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-getruntimeid).

## <a name="tree-views-and-navigation"></a>Visualizzazioni struttura ad albero e spostamenti

Gli elementi dell'interfaccia utente sullo schermo possono essere considerati come una struttura ad albero con il desktop come radice, le finestre delle applicazioni come elementi figlio immediati e gli elementi all'interno delle applicazioni come discendenti.

In Microsoft Active Accessibility, molti elementi dell'interfaccia utente irrilevanti per gli utenti finali vengono esposti nella struttura ad albero. Le applicazioni client devono esaminare tutti gli elementi dell'albero per determinare quali elementi sono significativi.

Le applicazioni client di automazione interfaccia utente visualizzano l'interfaccia utente tramite una visualizzazione filtrata. La vista contiene solo elementi che forniscono informazioni all'utente o che l'utente può interagire con. Sono disponibili visualizzazioni predefinite che includono solo elementi di controllo e solo elementi di contenuto e le applicazioni client possono definire visualizzazioni personalizzate. Automazione interfaccia utente semplifica la descrizione dell'interfaccia utente per l'utente e consente all'utente di interagire con le applicazioni.

In Microsoft Active Accessibility la navigazione tra gli elementi è spaziale, ad esempio per passare all'elemento che si trova a sinistra sullo schermo, logico, ad esempio, passare alla voce di menu successiva o all'elemento successivo nell'ordine di tabulazione in una finestra di dialogo o gerarchica, ad esempio, passare al primo elemento figlio in un contenitore o da un elemento figlio al relativo elemento padre. La navigazione gerarchica è complicata dal fatto che gli elementi figlio non sono sempre oggetti che implementano [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible).

In automazione interfaccia utente tutti gli elementi dell'interfaccia utente sono oggetti COM che espongono l'interfaccia [**IUIAutomationElement**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationelement) e supportano le stesse funzionalità di base. Dal punto di vista del provider, gli oggetti COM implementano un'interfaccia ereditata da [**IRawElementProviderSimple**](/windows/desktop/api/UIAutomationCore/nn-uiautomationcore-irawelementprovidersimple). Lo spostamento è principalmente gerarchico; ovvero dagli elementi padre agli elementi figlio e da un elemento di pari livello a quello successivo. Tuttavia, la navigazione tra gli elementi di pari livello presenta un elemento logico, in quanto può seguire l'ordine di tabulazione. Un client può spostarsi da qualsiasi punto iniziale, usando qualsiasi visualizzazione filtrata dell'albero, usando [**IUIAutomationTreeWalker**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationtreewalker). Un client può anche passare a elementi figlio o discendenti specifici usando [**IUIAutomationElement:: FindFirst**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-findfirst) e [**IUIAutomationElement:: FindAll**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-findall). Ad esempio, è facile recuperare tutti gli elementi di una finestra di dialogo che supportano un pattern di controllo specificato.

La navigazione nell'automazione dell'interfaccia utente è più coerente rispetto a Microsoft Active Accessibility. Alcuni elementi, ad esempio elenchi a discesa e finestre popup, vengono visualizzati due volte nella struttura ad albero di Microsoft Active Accessibility e la navigazione da tali elementi potrebbe produrre risultati imprevisti. È difficile implementare Microsoft Active Accessibility correttamente per un controllo Rebar. L'automazione interfaccia utente consente il riposizionamento e il riposizionamento dell'elemento padre, in modo che un elemento possa essere posizionato in qualsiasi punto dell'albero, nonostante la gerarchia imposta dalla proprietà di Windows.

## <a name="roles-and-control-types"></a>Ruoli e tipi di controlli

Microsoft Active Accessibility usa la proprietà accRole ([**IAccessible:: Get \_ accRole**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-get_accrole)) per recuperare una descrizione del ruolo dell'elemento nell'interfaccia utente, ad esempio [**il \_ dispositivo di \_ scorrimento del sistema dei ruoli**](object-roles.md) o l'oggetto [**\_ \_ MenuItem del sistema**](object-roles.md)del ruolo. Il ruolo di un elemento rappresenta l'indicazione principale della relativa funzionalità disponibile. L'interazione con un controllo viene eseguita utilizzando metodi fissi quali [**IAccessible:: accSelect**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-accselect) e [**IAccessible:: accDoDefaultAction**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-accdodefaultaction). L'interazione tra l'applicazione client e l'interfaccia utente è limitata alle operazioni che possono essere eseguite tramite [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible).

L'automazione dell'interfaccia utente, invece, separa il tipo di controllo dell'elemento, descritto dalla proprietà [**IUIAutomationElement:: CurrentControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_currentcontroltype) (o [**IUIAutomationElement:: CachedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_cachedcontroltype)), dalla relativa funzionalità prevista. La funzionalità è determinata dai pattern di controllo che sono supportati dal provider tramite la relativa implementazione di interfacce specializzate. I pattern di controllo possono essere combinati per descrivere il set completo di funzionalità supportate da un particolare elemento dell'interfaccia utente. Alcuni provider sono necessari per supportare un particolare pattern di controllo. Ad esempio, il provider per una casella di controllo deve supportare il pattern di controllo di [attivazione/](uiauto-implementingtoggle.md) disattivione. Altri provider sono necessari per supportare uno o più set di pattern di controllo. Un pulsante, ad esempio, deve supportare il pattern di controllo interruttore o [Invoke](uiauto-implementinginvoke.md) . Altri ancora non supportano alcun pattern di controllo. Ad esempio, un riquadro che non può essere spostato, ridimensionato o ancorato non dispone di pattern di controllo.

Automazione interfaccia utente supporta i controlli personalizzati, che sono identificati dalla costante [**\_ CustomControlTypeId di UIA**](uiauto-controltype-ids.md) e possono essere descritti dalla proprietà [**IUIAutomationElement:: CurrentLocalizedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_currentlocalizedcontroltype) (o [**IUIAutomationElement:: CachedLocalizedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_cachedlocalizedcontroltype)).

## <a name="states-and-properties"></a>Stati e proprietà

Gli elementi Microsoft Active Accessibility supportano un set di proprietà comune. Alcune proprietà, ad esempio accState, devono descrivere condizioni diverse, a seconda del ruolo dell'elemento. I server devono implementare tutti i metodi di [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible) che restituiscono una proprietà, anche le proprietà che non sono rilevanti per l'elemento.

Automazione interfaccia utente definisce proprietà aggiuntive, alcune delle quali corrispondono agli Stati di Microsoft Active Accessibility. Alcune proprietà sono comuni a tutti gli elementi, ma altre proprietà sono specifiche per i tipi di controllo e i pattern di controllo. Un provider di automazione interfaccia utente non deve implementare proprietà irrilevanti, ma può restituire un valore null per le proprietà che non supporta. Il servizio di base di automazione interfaccia utente può ottenere alcune proprietà dal provider di finestra predefinito e queste sono amalgamate con le proprietà implementate in modo esplicito dal provider.

Oltre a supportare molte più proprietà, l'automazione dell'interfaccia utente consente prestazioni migliori consentendo la memorizzazione nella cache delle proprietà.

## <a name="events"></a>Eventi

A differenza di Microsoft Active Accessibility, il meccanismo di eventi nell'automazione interfaccia utente non si basa sul routing degli eventi di Windows, che è strettamente collegato agli handle di finestra e non richiede l'impostazione di hook da parte dell'applicazione client. Le sottoscrizioni degli eventi possono essere ottimizzate a determinate parti dell'albero, non solo a determinati eventi. I provider possono anche ottimizzare la generazione di eventi tenendo traccia degli eventi di cui è in corso l'ascolto.

È inoltre più facile per i client recuperare gli elementi che generano eventi perché vengono passati direttamente al callback dell'evento. Le proprietà dell'elemento vengono automaticamente prelette, se è stata specificata una richiesta di cache quando il client ha sottoscritto l'evento.

## <a name="accessing-active-accessibility-properties-and-objects-from-ui-automation"></a>Accesso a proprietà e oggetti di Active Accessibility dall'automazione interfaccia utente

Una funzionalità chiave dell'automazione dell'interfaccia utente che non è disponibile in Microsoft Active Accessibility è la possibilità di recuperare più proprietà con una singola operazione tra processi.

I client Microsoft Active Accessibility esistenti possono sfruttare questa possibilità usando l'interfaccia [**IUIAutomationLegacyIAccessiblePattern**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) . Questa interfaccia rappresenta un *pattern di controllo* che espone le proprietà e i metodi di Microsoft Active Accessibility negli elementi dell'interfaccia utente. Quando si recuperano elementi, un'applicazione può richiedere che questo pattern di controllo e le relative proprietà vengano memorizzati nella cache.

[**IUIAutomationLegacyIAccessiblePattern**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) consente inoltre ai client di ottenere le proprietà di Microsoft Active Accessibility da elementi che non dispongono di supporto nativo per [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible).

Le modifiche apportate alle proprietà di un [**IUIAutomationLegacyIAccessiblePattern**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) non generano eventi di automazione interfaccia utente.
