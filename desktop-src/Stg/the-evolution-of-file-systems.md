# <a name="the-evolution-of-file-systems"></a><span data-ttu-id="14795-103">Evoluzione dei file System</span><span class="sxs-lookup"><span data-stu-id="14795-103">The Evolution of File Systems</span></span>

<span data-ttu-id="14795-104">Prima che i computer venissero sviluppati per funzionare su sistemi operativi su disco, ogni computer era progettato per eseguire un'unica applicazione proprietaria, che aveva un controllo completo ed esclusivo dell'intero computer.</span><span class="sxs-lookup"><span data-stu-id="14795-104">Before computers were developed to function on disk operating systems, each computer was built to run a single, proprietary application, which had complete and exclusive control of the entire machine.</span></span> <span data-ttu-id="14795-105">L'applicazione scrive i dati persistenti direttamente in un disco o in un tamburo inviando i comandi direttamente al controller del disco.</span><span class="sxs-lookup"><span data-stu-id="14795-105">The application would write its persistent data directly to a disk, or drum, by sending commands directly to the disk controller.</span></span> <span data-ttu-id="14795-106">L'applicazione è stata responsabile della gestione dei percorsi assoluti dei dati sul disco, assicurandosi che non sovrascriva i dati già esistenti.</span><span class="sxs-lookup"><span data-stu-id="14795-106">The application was responsible for managing the absolute locations of data on the disk, making sure that it was not overwriting already-existing data.</span></span> <span data-ttu-id="14795-107">Poiché una sola applicazione era in esecuzione nel computer in qualsiasi momento, questa attività non era troppo complessa.</span><span class="sxs-lookup"><span data-stu-id="14795-107">Since only one application was running on the computer at any time, this task was not too difficult.</span></span>

<span data-ttu-id="14795-108">L'avvento dei sistemi informatici che potevano eseguire più di un'applicazione richiedeva un meccanismo per garantire che le applicazioni non scrivessero i dati degli altri.</span><span class="sxs-lookup"><span data-stu-id="14795-108">The advent of computer systems that could run more than one application required a mechanism to ensure that applications did not write over each other's data.</span></span> <span data-ttu-id="14795-109">Gli sviluppatori di applicazioni hanno risolto questo problema adottando un unico standard per distinguere i settori del disco in uso da quelli che erano gratuiti, definendoli di conseguenza.</span><span class="sxs-lookup"><span data-stu-id="14795-109">Application developers addressed this problem by adopting a single standard for distinguishing disk sectors in use from those that were free by marking them accordingly.</span></span> <span data-ttu-id="14795-110">Nel tempo questi standard si uniscono a un sistema operativo su disco, che fornisce vari servizi alle applicazioni, incluso un file system per la gestione dell'archiviazione permanente.</span><span class="sxs-lookup"><span data-stu-id="14795-110">In time, these standards coalesced to become a disk operating system, which provided various services to the applications, including a file system for managing persistent storage.</span></span> <span data-ttu-id="14795-111">Con l'avvento di una file system, le applicazioni non devono più gestire direttamente il supporto di archiviazione fisica.</span><span class="sxs-lookup"><span data-stu-id="14795-111">With the advent of a file system, applications no longer had to deal directly with the physical storage medium.</span></span> <span data-ttu-id="14795-112">Al contrario, hanno semplicemente detto al file system di scrivere blocchi di dati sul disco e lasciare che i file system si preoccupino di come farlo.</span><span class="sxs-lookup"><span data-stu-id="14795-112">Instead, they simply told the file system to write blocks of data to the disk and let the file system worry about how to do it.</span></span> <span data-ttu-id="14795-113">Inoltre, il file system le applicazioni consentite di creare gerarchie di dati tramite un'astrazione nota come una directory.</span><span class="sxs-lookup"><span data-stu-id="14795-113">In addition, the file system allowed applications to create data hierarchies through an abstraction known as a directory.</span></span> <span data-ttu-id="14795-114">Una directory potrebbe contenere non solo file ma altre directory, che a loro volta potrebbero contenere i propri file e directory e così via.</span><span class="sxs-lookup"><span data-stu-id="14795-114">A directory could contain not only files but other directories, which in turn could contain their own files and directories, and so on.</span></span>

<span data-ttu-id="14795-115">Il file system ha fornito un unico livello di riferimento indiretto tra le applicazioni e il disco e il risultato è che ogni applicazione ha visualizzato un file come un singolo flusso contiguo di byte sul disco, anche se il file system stava effettivamente archiviando il file in settori non contigui.</span><span class="sxs-lookup"><span data-stu-id="14795-115">The file system provided a single level of indirection between applications and the disk, and the result was that every application saw a file as a single contiguous stream of bytes on the disk even though the file system was actually storing the file in discontiguous sectors.</span></span> <span data-ttu-id="14795-116">Il riferimento indiretto per le applicazioni ha liberato la necessità di tenere traccia della posizione assoluta dei dati in un dispositivo di archiviazione.</span><span class="sxs-lookup"><span data-stu-id="14795-116">The indirection freed the applications from having to track the absolute position of data on a storage device.</span></span>

<span data-ttu-id="14795-117">Attualmente, praticamente tutte le API di sistema per l'input e l'output di file forniscono applicazioni per la scrittura di informazioni in un file flat.</span><span class="sxs-lookup"><span data-stu-id="14795-117">Today, virtually all system APIs for file input and output provide applications for writing information into a flat file.</span></span> <span data-ttu-id="14795-118">Le applicazioni visualizzano questo file come un singolo flusso di byte che può aumentare fino a quando il disco non è pieno.</span><span class="sxs-lookup"><span data-stu-id="14795-118">Applications see this file as a single stream of bytes that can grow as large as necessary until the disk is full.</span></span> <span data-ttu-id="14795-119">Per un lungo periodo di tempo queste API sono sufficienti per le applicazioni per archiviare le informazioni persistenti.</span><span class="sxs-lookup"><span data-stu-id="14795-119">For a long time these APIs have been sufficient for applications to store their persistent information.</span></span> <span data-ttu-id="14795-120">Le applicazioni hanno innovazioni significative nel modo in cui si occupano di un unico flusso di informazioni per fornire funzionalità come i salvataggi "veloci" incrementali.</span><span class="sxs-lookup"><span data-stu-id="14795-120">Applications have made significant innovations in how they deal with a single stream of information to provide features like incremental "fast" saves.</span></span>

<span data-ttu-id="14795-121">Tuttavia, in un mondo di oggetti componente, l'archiviazione dei dati in un singolo file flat non è più efficiente.</span><span class="sxs-lookup"><span data-stu-id="14795-121">However, in a world of component objects, storing data in a single flat file is no longer efficient.</span></span> <span data-ttu-id="14795-122">Così come i file System sono emersi dalla necessità per più applicazioni di condividere lo stesso supporto di archiviazione, ora, gli oggetti componente richiedono un sistema che consenta loro di condividere l'archiviazione all'interno del framework concettuale di un singolo file.</span><span class="sxs-lookup"><span data-stu-id="14795-122">Just as file systems arose out of the need for multiple applications to share the same storage medium, so, now, do component objects require a system that allows them to share storage within the conceptual framework of a single file.</span></span> <span data-ttu-id="14795-123">Anche se è possibile archiviare gli oggetti separati usando l'archiviazione file flat convenzionale, se uno degli oggetti aumenta di dimensione o se si aggiunge semplicemente un altro oggetto, è necessario caricare l'intero file in memoria, inserire il nuovo oggetto e salvare l'intero file.</span><span class="sxs-lookup"><span data-stu-id="14795-123">Even though it is possible to store the separate objects using conventional flat-file storage, if one of the objects increases in size, or if you simply add another object, it becomes necessary to load the entire file into memory, insert the new object, and then save the whole file.</span></span> <span data-ttu-id="14795-124">Questo processo può richiedere molto tempo.</span><span class="sxs-lookup"><span data-stu-id="14795-124">This process can be extremely time-consuming.</span></span>

<span data-ttu-id="14795-125">La soluzione fornita da COM consiste nell'implementazione di un secondo livello di riferimento indiretto: un file system all'interno di un file.</span><span class="sxs-lookup"><span data-stu-id="14795-125">The solution provided by COM is to implement a second level of indirection: a file system within a file.</span></span> <span data-ttu-id="14795-126">Per l'archiviazione file flat è necessario che un'ampia sequenza di byte contigui sul disco venga modificata tramite un singolo handle di file con un unico puntatore di ricerca.</span><span class="sxs-lookup"><span data-stu-id="14795-126">Flat-file storage requires that a large contiguous sequence of bytes on the disk be manipulated through a single file handle with a single seek pointer.</span></span> <span data-ttu-id="14795-127">Al contrario, l'archiviazione strutturata COM definisce come trattare una singola entità di file system come raccolta strutturata di due tipi di oggetti, ovvero archivi e flussi, che fungono da directory e file.</span><span class="sxs-lookup"><span data-stu-id="14795-127">By contrast, COM structured storage defines how to treat a single file system entity as a structured collection of two types of objects — storages and streams — that act like directories and files.</span></span>
